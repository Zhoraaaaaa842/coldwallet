## Общая оценка: 7.5 / 10

Проект реализует корректную архитектуру холодного кошелька с air-gapped подписью.
Ниже — детальный анализ каждого компонента.

## 1. Генерация ключей (key_manager.py)

### ✅ Что хорошо

| Аспект | Реализация | Статус |
|--------|-----------|--------|
| Энтропия | 256 бит (24 слова BIP-39) | Безопасно |
| Деривация | BIP-44 (m/44'/60'/0'/0/0) | Стандарт |
| Библиотека мнемоники | `mnemonic` (референсная реализация BIP-39) | Безопасно |
| HD-кошелёк | `eth_account` от Ethereum Foundation | Доверенный |

### ⚠️ Рекомендации

1. **CSPRNG**: Генерация мнемоники использует `os.urandom()` внутри библиотеки `mnemonic` — это криптографически стойкий PRNG. Безопасно.
2. **Валидация мнемоники при импорте**: Реализована через `Mnemonic.check()` — проверяет контрольную сумму. Корректно.
3. **Очистка памяти**: Метод `clear()` перезаписывает ключ нулями перед удалением. Python не гарантирует физическое стирание из RAM (сборщик мусора может оставить копии), но это лучшая практика для Python. Для полной безопасности потребуется C-расширение с `SecureZeroMemory` / `explicit_bzero`.

### Уровень риска: НИЗКИЙ

## 2. Шифрование (key_manager.py — encrypt/decrypt)

### ✅ Что хорошо

| Параметр | Значение | Оценка |
|----------|---------|--------|
| Алгоритм | AES-256-GCM | Золотой стандарт |
| KDF | PBKDF2-HMAC-SHA256 | Проверенный |
| Итерации PBKDF2 | 600,000 | Выше OWASP минимума |
| Salt | 256 бит (случайный) | Безопасно |
| Nonce | 96 бит (случайный) | Стандарт GCM |
| AAD | Адрес кошелька | Защита целостности |
| Атомарная запись | os.replace() | Защита от corruption |

### ⚠️ Рекомендации

1. **PBKDF2 vs Argon2id**: PBKDF2 безопасен, но Argon2id устойчивее к GPU-атакам. Рекомендую миграцию на Argon2id в v2.0.
2. **Nonce reuse**: Каждое шифрование генерирует новый случайный nonce — безопасно. Но при 2^48 операциях с одним ключом есть теоретический риск коллизии. На практике нерелевантно.
3. **Версионирование**: Формат файла содержит `version: 1` — позволяет безболезненную миграцию криптографии.

### Уровень риска: НИЗКИЙ

## 3. Подпись транзакций (transaction.py)

### ✅ Что хорошо

- Поддержка EIP-1559 (Type 2) и Legacy транзакций
- Валидация всех параметров перед подписью
- Сериализация/десериализация для air-gapped переноса
- Подпись через `eth_account` (ECDSA secp256k1) — стандартная библиотека

### ⚠️ Рекомендации

1. **Replay protection**: `chain_id` включён в транзакцию — защита от replay-атак между сетями.
2. **Отображение деталей**: Утилита `sign_offline.py` показывает все детали TX перед подписью — защита от blind signing.
3. **Нет проверки баланса при подписи** — это нормально для air-gapped среды (нет сети), но десктоп-приложение должно проверять баланс перед созданием TX.

### Уровень риска: НИЗКИЙ

## 4. USB Storage (usb_manager.py)

### ✅ Что хорошо

- Чёткая файловая структура (pending/signed)
- Нет исполняемого кода в корне USB (только данные)
- Config в JSON (легко проверить)

### ⚠️ Рекомендации

1. **Шифрование на уровне диска**: Файлы `config.json` и TX в pending/signed не зашифрованы. Рекомендую шифрование USB через VeraCrypt или BitLocker в дополнение к шифрованию кошелька.
2. **Целостность TX**: Unsigned TX в pending/ не подписаны HMAC — теоретически можно подменить адрес получателя на USB. Решение: добавить HMAC с ключом, производным от пароля кошелька.
3. **Права доступа**: На Linux/macOS рекомендую `chmod 600` для wallet.vault.

### Уровень риска: СРЕДНИЙ

---

## 5. Сетевой модуль (eth_network.py)

### ✅ Что хорошо

- Используется ТОЛЬКО на онлайн-ПК (десктоп)
- Fallback через несколько публичных RPC
- Timeout 10 секунд (защита от зависания)

### ⚠️ Рекомендации

1. **Приватность**: Публичные RPC видят ваш IP и запрашиваемые адреса. Для анонимности используйте свою ноду или VPN.
2. **RPC манипуляция**: Вредоносный RPC может вернуть неверный баланс/nonce. Рекомендую кросс-проверку через 2+ ноды для критичных данных.
3. **SSL verification**: web3.py по умолчанию проверяет SSL-сертификаты — безопасно.

### Уровень риска: СРЕДНИЙ

---

## 6. Десктоп-приложение (main_window.py)

### ✅ Что хорошо

- Пароль запрашивается через `QInputDialog` с `EchoMode.Password`
- Сетевые запросы в отдельном потоке (QThread)
- `closeEvent` очищает ключи при закрытии
- Нет хранения пароля в памяти дольше необходимого

### ⚠️ Рекомендации

1. **Auto-lock**: Добавить таймер автоблокировки (например, 5 минут неактивности)
2. **Clipboard**: При копировании адреса данные остаются в буфере обмена. Рекомендую автоочистку через 60 секунд.
3. **Screen capture**: PyQt не защищает от скриншотов. На уровне приложения это нерешаемо.

### Уровень риска: НИЗКИЙ

## 7. Модель угроз

| Угроза | Защита | Статус |
|--------|--------|--------|
| Кража USB | AES-256-GCM + PBKDF2 (600k итераций) | ✅ Защищён |
| Keylogger на онлайн-ПК | Ключ никогда не покидает офлайн-среду | ✅ Защищён |
| Подмена TX на USB | ⚠️ Нет HMAC на pending TX | Рекомендуется |
| Malware на офлайн-ПК | Air-gap предполагает чистую ОС | ✅ By design |
| Brute-force пароля | 600k PBKDF2 итераций (~0.3 сек/попытка) | ✅ Защищён |
| Утечка из RAM | Перезапись нулями (ограничена Python GC) | ⚠️ Частично |
| RPC манипуляция | Нет кросс-проверки | ⚠️ Рекомендуется |
| Replay attack | chain_id в транзакции | ✅ Защищён |

## 8. Критические предупреждения

### 🔴 Это НЕ замена аппаратному кошельку

Ledger/Trezor имеют аппаратный Secure Element (SE), который:
- Физически изолирует приватный ключ
- Защищает от side-channel атак
- Имеет tamper-proof корпус

ColdVault — это **программный** холодный кошелёк. Он безопасен при:
- Использовании на чистой офлайн-машине
- Сильном пароле (12+ символов)
- Физической безопасности USB

### 🔴 Мнемоника = полный доступ

Кто имеет 24 слова — владеет кошельком. Храните ТОЛЬКО на бумаге, в сейфе.

## Заключение

ColdVault ETH реализует **корректную криптографию** и **правильную архитектуру** air-gapped холодного кошелька. Основные риски связаны с физической безопасностью USB и ограничениями Python при работе с памятью. Для хранения значительных сумм рекомендуется аппаратный кошелёк (Ledger, Trezor). Для умеренных сумм и обучения — ColdVault обеспечивает достаточный уровень безопасности при соблюдении рекомендаций.
